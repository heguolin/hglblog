# 给 AI 看板娘装上「记忆」——流萤 RAG 系统从零到一的踩坑实录

> 一个 0.6B 小模型，一台 4C4G 服务器，从幻觉满天飞到准确回答博客知识的全过程。

---

## 为什么做这个

博客上线后，右下角的流萤看板娘成了访客最爱点的地方。但她有个致命缺陷：**她完全不知道博客里有什么**。你问"有什么文章？"，她会用训练数据里学来的角色语气编一段——语气很"流萤"，答案全是假的。

大模型没有自己数据的记忆，想让她"知道"博客内容，最成熟的路子是 **RAG（检索增强生成）**——用户提问时先从知识库搜相关内容，再把搜到的内容塞给模型当"参考书"，让它对着参考书回答。

听起来很简单。实际上在一台 4C4G CPU 服务器上做这件事，踩了整整一周的坑。

---

## 第一关：选技术栈

最初的计划是这样：

```
用户 → Nginx → Python RAG 微服务 → Qwen3-0.6B 模型
                   ↓
               Milvus 向量库 ← PostgreSQL（博客内容）
```

Embedding 模型选了 BGE-small-zh（中文检索最强的小模型，约 100MB），向量库选了 Milvus。

计划很美好，一上服务器就炸了。

**坑1：Milvus 不兼容。** 服务器是 CentOS 7，Milvus v2.5.0 需要新版 glibc，容器直接报 `ERROR: ld.so: object from LD_PRELOAD cannot be preloaded` 然后无限重启。折腾两小时，试了改内核参数、降版本、甚至手动编译——全都不行。

**解决：换 Chroma。** Chroma 是 Python 原生的向量数据库，`pip install chromadb` 就装好，进程内运行用 SQLite 存索引，不需要独立容器。省了 800MB 内存，还少一个需要维护的服务。对于几百篇博客文章的量级，绰绰有余。

---

## 第二关：模型跟搜索抢风头

知识库搭好了，检索也通了。测试时发现：**微调过的流萤模型不听话。**

她的训练数据里有"你是流萤，星核猎手成员，要用温柔的语气说话"这类角色设定。注入博客内容后问她"博客里有什么文章？"，她回答：

> "在匹诺康尼的废墟上，我看过最珍贵的风景。"

……压根没看检索结果。你把参考书塞到她面前，她偏要按剧本演。

试了三种 prompt：

1. **柔和版**：请根据以上知识回答 → 直接无视，继续角色扮演
2. **强硬版**：【重要指令】你必须根据以下内容…禁止编造！→ 听话了，但变成冷冰冰答题机器，"流萤"感全没了
3. **折中版**：用流萤的语气提到以下信息 → 时而听话时而放飞

整整调了一下午 prompt，最好的结果也不可靠。0.6B 小模型的天花板就在这里——它能做好角色扮演，也能做好知识问答，但**不能同时兼顾**。

**最终方案：路由分流。**

- 日常聊天 → 微调模型（保持流萤角色感）
- 博客知识 → 不调模型，**直接返回检索到的原文片段**

知识模式下跳过 LLM，零幻觉，延迟从 27 秒降到 0 毫秒。用户问"怎么部署 Docker？"，直接贴博客里那篇教程的段落。用户问"今天天气"，返回"唔…博客里没有相关内容"。

这个取舍很关键：**不是所有问题都需要 AI 生成答案。搜索就好好做搜索。**

---

## 第三关：内存与 OOM

双模型方案（角色模型 + 基座模型各跑一个实例）写到一半放弃了。两个 Qwen3-0.6B 实例并行需要 ~2.7GB 内存，加上 PG、Redis、Chroma、BGE，总共 ~4.2GB，OOM killer 杀了五次。

五次之后学会了一件事：**在 4G 内存上做 RAG，每一个字节都要计较。**

最终方案：
- 只跑一个模型实例（微调版）
- Prompt 切换代替模型切换（闲聊用角色 prompt，知识模式直接返回片段）
- BGE 模型常驻但只用一份实例
- batch_size 从 32 降到 4（入库时每次处理 4 篇文章，慢但不会 OOM）

顺便学到了 Docker 磁盘管理：`docker system prune -af` 能救急。

---

## 第四关：HuggingFace 被墙

Docker build 时有个 `RUN python -c "SentenceTransformer('BAAI/bge-small-zh')"` 预下载模型。在国内服务器上跑了十分钟后报错：`Network is unreachable`。

**解决：离线部署。** 在本机（能翻墙）下载模型，`tar` 打包，`scp` 上传到服务器。Dockerfile 去掉预下载步骤，改成 `-v ./models:/app/models:ro` 挂载本地模型目录。`EMBEDDING_MODEL=/app/models/bge-small-zh` 环境变量指过去。

这个模式也适用于其他需要模型的场景——先本地下载，再上传，不在服务端做任何网络下载。

---

## 最终架构

```
用户 → Nginx → Python RAG (:8002) ─┬─ 闲聊 → Qwen3-0.6B 微调版 (:8001)
                                    │
                                    └─ 知识模式 → Chroma 检索 → 直接返回片段（不调模型）
                                          ↑
                                   BGE-small-zh 向量化
                                          ↑
                                  PostgreSQL ← 离线入库脚本
```

| 组件 | 选择 | 原因 |
|------|------|------|
| 向量库 | Chroma | CentOS 兼容，进程内，省内存 |
| Embedding | BGE-small-zh（本地） | 中文最好，100MB，不花钱 |
| 模型部署 | 离线下载 + volume 挂载 | 避开 HF 被墙 |
| 角色聊天 | Qwen3-0.6B + LoRA 微调 | 流萤人设，CPU 可跑 |
| 知识问答 | 不调模型，直接返回检索结果 | 零幻觉，零延迟 |
| 路由 | 前端手动切换 + 后端关键词/模式路由 | 用户可控，不猜意图 |

## 感受

做这个实验最大的体会：**小模型做 RAG，别指望它同时做好角色和知识，分而治之才是正道。**

第二个体会：**提示词工程在大模型上有用，在小模型上不靠谱。** 花了大量时间调 prompt，还不如改架构——知识模式直接不走模型。最好的 prompt 是不需要 prompt。

第三个体会：**4C4G 服务器能做的事比想象中多。** 前提是放弃"大而全"，接受"小而精"。Chroma 替代 Milvus、单模型替代双模型、直接返回替代生成——每个决策都省了内存，最终勉强跑在 4GB 内。

还有一个意外的收获：前端加的模式切换开关（流萤 / 自动 / 知识），让用户自己选，既省了后端检测逻辑，也给了用户控制感。设计上的小改动比模型层的复杂优化更有效。
